# 测试用例规划文档

> 本文档面向人类阅读者，说明 E2E 测试的整体思路、分类原则、每个测试点的设计意图，
> 以及边界条件的考量逻辑。代码实现见各 `.spec.ts` 文件。

---

## 一、整体设计思路

### 测试目标

本项目不追求"全绿"——绿色只证明代码按现有逻辑运行，不能证明业务正确。

测试分两类目标：

1. **行为验证**：用户能完成核心操作路径（登录→入库→发货→报表）
2. **缺陷发现**：从业务语义出发，主动构造边界场景，让真正的 Bug 以红色测试呈现

`[已知缺陷]` 标注的用例**预期失败**，它们是活的 Bug 报告，不是写坏的测试。
当主项目修复后，对应测试改为验证正确行为（移除 `[已知缺陷]` 标注），全套变绿。

### 测试分层

```
UI 端到端（Playwright Browser）
  ↓ 验证用户实际操作路径、页面跳转、视觉样式
API 级别（Playwright Request Fixture）
  ↓ 验证业务逻辑、输入校验、权限控制、边界条件
```

大量边界测试直接走 API，原因：
- 速度快（无浏览器渲染开销）
- 断言精确（HTTP 状态码、JSON 字段值）
- 不依赖 UI selector 的稳定性

### 执行顺序与状态隔离

测试使用同一个真实 SQLite 数据库（非 mock），单 worker 顺序执行。
目录名决定执行顺序（字母序）：

```
auth/ → inventory/ → orders/ → products/ → rbac/ → regression/ → reports/
```

**regression/ 排在 rbac/ 之后**：边界测试会创建额外批次和订单，
如果排在前面会污染 dashboard 等依赖精确种子数量的断言。

每次完整运行前须执行 `npm run seed` 重置数据库。

## 二、测试分类说明

### `auth/` — 认证流程

**为什么单独一类**：认证是所有功能的门卫，任何认证失效会导致整个系统不可用。

**覆盖点**：

| 用例 | 测试意图 |
|------|---------|
| admin/staff 正常登录跳转首页 | 验证两个角色的完整登录链路 |
| 密码错误显示错误提示 | 验证失败反馈，防止静默失败 |
| 未登录访问受保护路由被重定向 | 验证 auth guard 生效 |
| 重定向携带 `redirect` 参数 | 验证登录后能回到原目标页 |

**边界考量**：auth guard 只测"未登录"场景，角色级别的路由拦截在 `rbac/` 测试。

---

### `inventory/` — 库存核心功能

#### `dashboard.spec.ts` — 保质期预警看板

**为什么重要**：看板是运营的第一视角，数字错误会导致错误决策（如忽视过期库存）。

| 用例 | 测试意图 |
|------|---------|
| 过期/临期/健康数量正确 | 验证 `classify_expiry_status` 分类与计数逻辑 |
| 点击卡片跳转到对应筛选页 | 验证卡片可交互且参数传递正确 |

**边界考量**：种子数据精确覆盖三种状态各自的批次，数量断言使用精确值（非 ≥ 1）。
今天到期的边界（`expiry_date == today`）在 `regression/expiry-boundary` 专项测试。

#### `inbound.spec.ts` — H5 入库流程

**为什么重要**：入库是库存的唯一来源，流程中断或数据写错会导致库存不准。

| 用例 | 测试意图 |
|------|---------|
| 三步引导展示 | 验证 UI 流程结构完整 |
| 条码识别 / HB 编码识别 | 验证两种商品查找方式均可用 |
| 未知编码显示提示 | 验证错误分支有明确反馈，不静默失败 |
| 完整入库流程（识别→填写→提交） | 验证主流程端到端可走通 |
| 采购单位换算（2盒→12支） | 验证 `conversion_rate` 换算逻辑正确写入 |

**边界考量**：每个入库测试使用不同商品（HB002/HB003/HB001），避免批次合并干扰断言。
换算测试验证的是"2盒×6=12支"，而非仅验证"入库成功"。

#### `stock-report.spec.ts` — 库存到期报表

**为什么重要**：库存报表是运营日常核查过期风险的主要工具，筛选错误等于盲操作。

| 用例 | 测试意图 |
|------|---------|
| 全部/过期/临期/健康筛选 | 验证四种过滤器均有效 |
| 红/橙/绿背景颜色 | 验证视觉预警与状态一致（CSS class） |
| 批次卡片展示到期日与库存数字 | 验证关键字段渲染 |

**边界考量**：颜色测试直接断言 CSS class（`bg-red-50` 等），不依赖截图对比，更稳定。

---

### `orders/` — 订单发货核销

**为什么重要**：发货核销是不可逆操作，会永久扣减库存，任何异常都影响数据准确性。

| 用例 | 测试意图 |
|------|---------|
| 种子订单以"待发货"出现在列表 | 验证种子数据与 UI 渲染一致 |
| 点击发货→状态变为已发货 | 验证核销主流程及状态流转 |
| 通过表单同步新订单 | 验证订单同步入口可用 |

**边界考量**：发货后不检查库存精确数值（依赖跨测试状态）；
重复发货、不存在订单等异常场景在 `regression/fifo` 测试。

---

### `products/` — 商品中心

**为什么重要**：商品档案是库存、入库、订单的基础数据，档案错误会级联影响所有模块。

| 用例 | 测试意图 |
|------|---------|
| 列表显示所有商品 | 验证基础渲染 |
| 按名称/编码/条码搜索 | 验证三种搜索维度均有效 |
| 回车触发搜索 | 验证键盘交互（不只是点按钮） |
| 展开建档表单 / 创建成功 | 验证新建商品完整流程 |
| admin 看到导入按钮 / staff 不看到 | 验证前端 RBAC 显示逻辑 |
| 详情页：商品信息 + 批次列表 | 验证详情页数据正确加载 |
| 三个库存卡片显示非负整数 | 验证 UI 数值合法性（不显示 null/NaN） |
| 总库存 ≥ 可售库存（FIFO 不变式） | 通过 API 断言数学不变式，与 UI 无关 |
| 单位换算展示（1盒=6支） | 验证 conversion_rate 正确渲染 |
| 不存在的商品显示错误状态 | 验证 404 分支有视觉反馈 |
| 列表点击"查看详情"进入详情页 | 验证导航链路 |

**边界考量**：库存数值不断言精确值（会被其他测试改变），改用不变式断言（`total >= sellable`）。

---

### `rbac/` — 前端路由权限控制

**为什么单独一类**：前端 RBAC 和后端 RBAC 是两个独立层，需分开验证。
本类只测前端路由 guard；后端 API 权限在 `regression/rbac-api` 测试。

| 用例 | 测试意图 |
|------|---------|
| staff 访问 /settings 被重定向 | 验证 admin-only 路由的 guard 生效 |
| staff 访问 /finance 被重定向 | 同上，另一个受保护路由 |
| staff 侧边栏不显示"设置"入口 | 验证 UI 元素随角色隐藏（减少误操作） |
| admin 可访问 /settings / /finance | 验证 admin 不被误拦截 |
| admin 侧边栏显示"设置"入口 | 验证 admin 看到完整菜单 |

---

### `regression/` — 边界条件与已知缺陷

这是本项目最核心的部分，从**业务语义**出发设计，而非贴着实现代码写。

#### `fifo.spec.ts` — FIFO 库存分配逻辑

FIFO（先进先出）是系统的核心库存策略，错误的分配顺序会导致新货先发、旧货过期。

| 用例 | 测试意图 | 边界设计 |
|------|---------|---------|
| 跨批次分配时优先消耗最早到期批次 | 验证 FIFO 排序正确 | 动态读取第一批次可用量，预约量 = 可用量+1，强制触发跨批次 |
| 超出可售库存返回 409 | 验证超额预占被拒绝 | 动态读取 sellable_stock，用 sellable+1 触发 |
| 精确耗尽库存后再预约 1 个返回 409 | 验证库存耗尽临界点 | 先用 sellable 清空，再请求 1 个 |
| 重复发货同一订单返回 409 | 验证幂等保护 | 创建订单→发货→再次发货，第二次必须失败 |
| 不存在的订单 ID 返回 404 | 验证错误输入处理 | 用 999999 这种不可能存在的 ID |
| 未建档商品预约返回 404 | 验证商品不存在时的错误处理 | — |
| 未知渠道/SKU 同步订单返回 409 | 验证渠道映射不存在时的错误处理 | — |

#### `inbound-edge.spec.ts` — 入库接口输入校验

输入校验是数据质量的第一道防线，缺失校验会导致脏数据写入。

| 用例 | 测试意图 | 边界设计 |
|------|---------|---------|
| 缺少 batch_no 返回 400 | 必填字段校验 | 不传 batch_no 字段 |
| 缺少 expiry_date 返回 400 | 必填字段校验 | 不传 expiry_date 字段 |
| quantity=0 返回 400 | 零值校验 | 边界值：0 是合法整数但业务不允许 |
| quantity 为负数返回 400 | 负值校验 | -5，非法业务值 |
| cost 为负数返回 400 | 负值校验 | -1.0，成本不能为负 |
| 未建档商品返回 404 | 外键校验 | hb_code 指向不存在的商品 |
| 日期格式非 ISO 返回 400 | 格式校验 | `31/12/2030` vs 正确的 `2030-12-31` |
| 批次合并成本覆盖验证（BUG-01 修复验证） | 验证加权平均成本计算正确 | 动态读取入库前成本/数量，计算期望加权平均，断言修复后结果吻合 |

**边界设计说明**：测试先 GET 读取当前成本和数量，动态计算期望加权平均值，
不依赖种子的精确初始值，在多次运行时均能稳定验证。

#### `expiry-boundary.spec.ts` — 到期日分类边界

`classify_expiry_status` 的分界点在 `today`（`<=` today 为 expired）和 `today+30`（小于等于）。
这两个点是 off-by-one 错误的高发区，BUG-02 已修复（`<` 改为 `<=`）。

| 用例 | 测试意图 | 边界设计 |
|------|---------|---------|
| today-1 → expired（基准） | 确认 expired 区间基础正确 | — |
| today+30 → warning（包含边界） | 验证 ≤ today+30 的包含语义 | 精确的第 30 天 |
| today+31 → healthy（刚超出窗口） | 验证 > today+30 落入 healthy | 第 31 天，刚好越过边界 |
| today → expired（BUG-02 修复验证） | 验证今天到期正确归类 expired | 精确边界：修复后 `<=` today，应为 expired |

所有日期在运行时动态计算（`dateOffset(n)`），测试在任何日期运行均有效。
使用专属商品 `HBBD` 存放边界批次，避免污染种子数据的精确计数。

#### `order-idempotency.spec.ts` — 订单同步幂等性

电商平台 webhook 重推是常见场景。BUG-03 修复方案：调用方传入 `external_order_no`，
重复调用返回同一订单，不重复预占库存（不传则保持兼容旧行为）。

| 用例 | 测试意图 | 边界设计 |
|------|---------|---------|
| 携带 `external_order_no` 重复 sync → 返回同一订单 | 验证幂等性修复（BUG-03）| 两次调用相同 `external_order_no`，断言 order.id 一致 |
| 不同 `external_order_no` → 独立创建两笔订单 | 验证不同订单不误拦截 | 对照组 |
| 不同 channel+SKU 各自创建独立订单 | 验证正常多渠道场景 | 对照组 |

#### `barcode-collision.spec.ts` — 条码唯一性

条码是物理世界与系统的唯一桥梁，碰撞会导致入库失效。BUG-04 已修复（API 层强制唯一）。

| 用例 | 测试意图 | 边界设计 |
|------|---------|---------|
| 相同条码创建第二个商品返回 409（BUG-04 修复验证） | 验证唯一约束生效 | 创建同条码第二商品，断言 409 |
| 通过条码入库精确命中唯一商品 | 验证唯一条码入库路由正确 | 唯一条码入库，断言命中正确 hb_code |
| 不同商品不同条码可以共存 | 正常场景对照组 | 确保唯一约束不误拦截正常创建 |

#### `rbac-api.spec.ts` — 后端 API RBAC 一致性

前端 RBAC 只影响页面访问，后端 API 的权限控制才真正决定安全边界。
BUG-05 / OBS-01 / OBS-02 均已修复，所有测试验证修复后正确行为。

| 用例 | 测试意图 | 类型 |
|------|---------|------|
| `GET /api/v1/inventory/test` 返回 404（OBS-01 修复验证） | 调试端点已从生产环境移除 | 安全修复验证 |
| staff 调用 `POST /products` 返回 403（BUG-05 修复验证） | 商品创建已改为 admin-only | 权限修复验证 |
| staff 调用 `orders/fulfill` 返回 403（OBS-02 修复验证） | fulfill 已改为 admin-only | 权限修复验证 |
| staff 调用 products/import 返回 403 | 验证 admin-only 端点正常拦截 | 保护性验证 |
| staff 调用 channel-mappings 返回 403 | 同上 | 保护性验证 |
| staff 调用 reports/export 返回 403 | 同上 | 保护性验证 |

### `reports/` — 报表导出

**为什么放最后**：reports 测试只做 HTTP 层断言（Content-Disposition、状态码），
不关心报表数据内容，受前面测试的状态影响最小，排在最后最安全。

| 用例 | 测试意图 | 边界设计 |
|------|---------|---------|
| admin 下载 CSV，Content-Disposition 正确 | 验证文件下载链路完整 | 检查 header，不验证文件内容 |
| admin 下载 xlsx | 验证另一种格式 | MIME type 断言 |
| staff 返回 403 | 验证报表为 admin-only | — |
| 未登录返回 401 | 验证无 token 被拒绝 | — |
| format=json 返回 400 | 验证非法参数校验 | json 是看似合理但不支持的格式 |
| ledger-export 同上覆盖 | 账本导出接口独立验证 | 包含 balance_scope 参数的合法/非法值 |

## 四、故意不测的内容

| 内容 | 原因 |
|------|------|
| 并发/竞态条件 | Playwright 单 worker 无法模拟真正并发；需压测工具另行测试 |
| 批量导入 CSV 流程（UI） | Stage 3 规划中，依赖文件上传交互，复杂度较高 |
| 精确库存数值断言 | 跨测试状态累积，改用不变式（total ≥ sellable）代替 |
| 报表文件内容正确性 | 内容变化频繁；Content-Disposition + MIME type 已足够验证下载链路 |
| 数据库直接写入 | 所有操作通过 API/UI 进行，保持黑盒测试立场 |
